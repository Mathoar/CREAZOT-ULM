Problèmes identifiés

5 problèmes bloquants

1
Prompt VAPI statique et vide
buildAssistantPrompt() ne contient aucun catalogue de circuits. Le $clientId est reçu mais jamais utilisé. Le LLM ne sait pas quelles prestations existent sans appeler list_services à chaque conversation. Aucune info sur les horaires, le fuseau, les règles métier du club.
2
Résolution de circuit incohérente
check_availability accepte code OU sous-chaîne du nom (souple). confirm_reservation accepte uniquement le code exact (strict). Le LLM vocal ne mentionne pas les codes aux clients → échec potentiel à la confirmation.
3
Email IA : pas de création automatique de réservation
ReservationAiService mène au statut awaiting_club mais ne crée jamais la Reservation. AiReservationController::validateReservation crée sans vérifier isSlotAvailable → risque de double booking.
4
CRON email non configuré
La commande app:poll-inbound-emails existe mais aucun cron ne l'appelle. Aucun client n'a d'aiReservationEmail ni de credentials IMAP configurés.
5
Aucune personnalisation par client
Le prompt est identique pour tous les clubs (seul le nom change). Aucune consigne spécifique, aucun circuit, aucune règle métier injectée. Chaque club a des prestations, horaires, qualifications complètement différentes.

Problèmes de cohérence métier

!
Statut incompatible
VAPI crée les réservations avec statut confirmé mais la BDD Planetair utilise VALIDATED, WAITING, WEATHER_REPORT, PASSENGER_CANCEL, etc. Le statut confirmé n'existe nulle part dans les données existantes. Risque : la réservation n'apparaît pas dans les filtres / vues du calendrier.

Solution : utiliser WAITING pour les créations IA (en attente de validation admin) ou VALIDATED si auto-confirmé.
!
Code réservation jamais généré
100% des réservations Planetair (1710/1710) ont un code au format RESA-{timestamp}-{random}. Les assistants IA ne génèrent pas ce code → la réservation est "orpheline" sans référence traçable.

Solution : générer automatiquement le code à la création.
!
Email de confirmation non envoyé
Planetair a hasEmailConfirmation = true avec un message de confirmation et un sender contact@planetair974.com. Mais quand VAPI crée une réservation, aucun email de confirmation n'est envoyé au client.

Solution : déclencher l'envoi du mail de confirmation après création (si module actif + email client disponible).
!
Options / combinaisons ignorées
Planetair a 2 options (Porte Photos 20€, Porte Photos Offerte 0€) et 5 combinaisons. Les assistants ne proposent jamais d'options — le prix final peut être faux.

Solution : intégrer les options dans le prompt et le tool-call confirm_reservation.
!
Bons cadeaux non gérés
Planetair a des bons cadeaux actifs (codes comme 8ll0ky1yl945) liés à des circuits spécifiques. Un client peut appeler/écrire avec un code cadeau — aucun des assistants ne gère ce cas.

Solution : ajouter un outil check_gift_code qui vérifie la validité du bon et auto-remplit circuit + prix.
!
Origine non tracée
Planetair a 13 origines (Le Routard, Web, Bouche à oreille, etc.) avec des remises associées. Les réservations IA devraient être auto-taguées avec une origine dédiée ("Assistant IA Vocal" / "Assistant IA Email").

Solution : créer une origine auto par canal, ou paramétrer une origine par défaut dans le builder.

Architecture cible

AssistantContextBuilder — Service central

Un service unique qui construit dynamiquement le contexte métier complet pour chaque client, utilisable par les deux canaux (VAPI vocal + Email Kimi).

┌─────────────────────────────────────────────────────────────┐ │ AssistantContextBuilder │ │ │ │ buildContext(Client) → array │ │ ├── Circuits du client (nom, code, prix, durée, nature) │ │ ├── Horaires d'ouverture (minH / maxH + timezone) │ │ ├── Qualifications & contraintes (encadrant, landing) │ │ ├── Options / combinaisons (porte-photos, etc.) │ │ ├── Infos club (nom, tél, email, adresse, aérodrome) │ │ └── Consignes personnalisées (champ libre admin) │ │ │ │ buildPrompt(array $context, string $channel) → string │ │ → Prompt structuré adapté au canal (vocal / email) │ └────────────────────┬────────────────────┬───────────────────┘ │ │ ┌──────────▼──────────┐ ┌──────▼──────────────┐ │ VapiService │ │ ReservationAi │ │ │ │ Service │ │ createAssistant() │ │ │ │ updateAssistant() │ │ buildSystemPrompt() │ │ │ │ │ │ Prompt = builder │ │ Prompt = builder │ │ .buildPrompt(ctx, │ │ .buildPrompt(ctx, │ │ 'voice') │ │ 'email') │ └─────────────────────┘ └─────────────────────┘

Ce que le builder injecte

  • Catalogue complet des circuits avec prix, durées, codes
  • Table de correspondance code ↔ nom (pour résolution fiable)
  • Horaires d'ouverture du club + fuseau horaire
  • Natures de vol et qualifications requises par circuit
  • Options et combinaisons disponibles (tarifs groupés)
  • Coordonnées du club (nom, tél, email, adresse)
  • Consignes personnalisées de l'admin (champ libre)

Ce que ça résout

  • Prompt pauvre → catalogue complet injecté
  • Pas de personnalisation → contexte dynamique par client
  • Résolution incohérente → table code/nom dans le prompt
  • Horaires invisibles → plages horaires dans le prompt
  • Options ignorées → options décrites
  • Règles métier cachées → qualifs, encadrant documentés
  • Duplication → un seul builder, deux canaux
